Auth Logic Test Suite Overview

Architectural Approach to Testing

The project's testing strategy mirrors its decoupled architecture, which separates state management (XState) from side effects (API calls). This results in multiple layers of tests:

  • State Machine Logic (authMachine.test.ts): Focuses on testing the user-facing flows and application logic, ensuring the system transitions between states correctly in response to events.
  • Data & API Repository (AuthRepository.test.ts): Focuses on testing the implementation details of data fetching, API communication, session management (JWTs), and error handling.
  • Repository Error Handling (AuthRepository.error.test.ts): Focuses on testing edge cases and error scenarios in the repository layer.
  • Error Handling Utilities (errorHandler.test.ts): Focuses on testing the error handling functions used across the auth system.
  • Utility Functions (safeExtractUtils.test.ts): Tests for safe data extraction utilities.
  • Safety Utilities (safetyUtils.test.ts): Tests for various safety utility functions used throughout the auth system.

This separation allows for robust, isolated testing. The state machine can be tested without making real network requests, and the repository can be tested without needing to manage application state.

I. State Machine Flow Tests: authMachine.test.ts

These tests validate the core application logic by simulating user interactions and asserting state transitions. The AuthRepository is mocked, allowing the tests to focus solely on the machine's behavior.

Test Hierarchy:

  • describe("Auth Machine")
    • Initialization:

      Checks the initial state when the app starts.

      • test("should move to unauthorized if no session")
      • test("should move to authorized if session exists")
    • Login Flow:

      Tests the complete login sequence.

      • test("should handle success"): User logs in, machine transitions to authorized.
      • test("should return to idle with an error on failure"): API fails, machine enters a failure state with an error message.
      • test("should fall back to default error message when failure lacks detail"): Handles failures without detailed messages.
    • Register Flow:

      Tests the user registration sequence with OTP verification.

      • test("should handle full OTP-based registration flow"): Complete flow from form submission to login.
      • test("should return to form with an error when register fails"): Handles registration failures.
      • test("should show default error when register rejects without payload"): Handles empty error responses.
      • test("should surface OTP errors"): Properly handles OTP verification failures.
      • test("should ignore VERIFY_OTP when email context is missing"): Guards against invalid transitions.
      • test("should complete registration with empty password when none stored"): Handles missing password scenarios.
    • Forgot Password Flow:

      Tests the entire multi-step password reset process with OTP verification.

      • test("should handle the full forgot password flow"): Complete multi-step flow from email submission to password reset.
      • test("should handle request failure"): Handles initial email submission failures.
      • test("should handle OTP verification failure"): Handles OTP validation failures.
      • test("should ignore VERIFY_OTP when email context is missing"): Prevents invalid OTP transitions.
      • test("should ignore RESET_PASSWORD when no action token exists"): Prevents invalid password reset transitions.
    • Navigation & Logout:

      Ensures smooth transitions between UI states and a clean logout.

      • test("should navigate between sub-states"): Checks navigation between login and register forms.
      • test("should logout and clear session"): Verifies that a LOGOUT event clears user data and returns to the unauthorized state.
    • Branch Coverage Tests:

      Additional tests to ensure full branch coverage.

      • test("should handle checkSession error"): Handles session check failures.
      • test("should handle error in login with null error object"): Handles null error objects.
      • test("should handle error in register with null error object"): Handles null error objects in registration.
  • describe("resolveRegistrationPassword")
    • test("returns provided password when non-empty string"): Returns valid password string.
    • test("returns empty string when password is missing or blank"): Handles missing/empty passwords.

II. Repository & API Tests: AuthRepository.test.ts

These tests validate the data layer, ensuring that the application correctly interacts with the backend API and manages local session storage. axios and IStorage are mocked to simulate network and storage operations.

Test Hierarchy:

  • describe("AuthRepository")
    • Login & Register:

      Verifies that API calls are made correctly and session tokens are stored.

      • test("should call API, save session, and return tokens on login success")
      • test("should register user via OTP-init endpoint without touching storage")
      • test("should surface server errors when login fails")
      • test("should surface fallback message when register fails without server message")
      • test("should throw network error message when no response payload exists")
      • test("should throw a generic error if error has no message")
    • Check Session (JWT Logic):

      This is a critical set of tests for session integrity.

      • test("should return null if no token exists")
      • test("should skip API and LOGOUT if token is LOCALLY expired"): Crucially, this prevents unnecessary network requests for expired tokens.
      • test("should proceed to API if token is LOCALLY valid")
      • test("should drop non-string refresh tokens when reading session")
      • test("should keep refresh token when stored session includes string value")
      • test("should LOGOUT if token is locally valid but SERVER rejects it (401)"): Handles cases where a session is revoked server-side.
      • test("should return null without logout when server responds with non-401 error")
      • test("should handle network errors without response object")
      • test("should handle legacy plain string tokens gracefully")
      • test("should handle malformed tokens gracefully (treat as invalid)")
      • test("should return null when stored JSON lacks accessToken")
    • Request Password Reset:

      Tests the initial OTP request functionality.

      • test("should call OTP request endpoint")
      • test("should throw when OTP request fails")
    • Verify OTP and Reset Password:

      Tests OTP verification and password reset completion.

      • test("should call verify-otp endpoint and return action token")
      • test("should throw when verify-otp fails")
      • test("should call password reset completion endpoint")
      • test("should throw when password reset completion fails")
    • Complete Registration:

      Tests the final registration step with action token.

      • test("should call complete registration endpoint")
      • test("should throw when completion fails")
    • Logout:

      Tests session cleanup.

      • test("should remove token from storage")
    • Retry Logic:

      Tests automatic retry mechanism for transient server errors.

      • test("should retry on 503")
    • Response Interceptor:

      Tests the response handling for the axios client.

      • test("should return response as-is for success handler")
      • test("should not retry when error is non-retryable")
      • test("should reject immediately when config is missing")
    • isTokenExpired:

      Tests the JWT expiration detection.

      • test("returns false when token lacks exp so server can decide")
      • test("uses Buffer fallback when atob is unavailable")
      • test("returns true when payload cannot be parsed")
      • test("returns false when neither atob nor Buffer are available")
      • test("returns true when token is missing required segments")
    • Refresh Token:

      Tests token refresh functionality.

      • test("should refresh access token using refresh token")
      • test("should throw error when refresh token request fails")
      • test("should throw error when no current session exists during refresh")
    • Check Session with Expired Tokens:

      Comprehensive tests for session handling with expired tokens.

      • test("should logout and return null when token is expired but no refresh token is present")
      • test("should attempt refresh when 401 occurs and refresh token is available")
      • test("should logout when 401 occurs and refresh token is not available")
      • test("should logout when token is expired and refresh attempt fails")
      • test("should logout when 401 occurs and refresh attempt fails")
      • test("should return null without logout when non-401 error occurs")
      • test("should return null without logout when non-axios error occurs")
      • test("should logout and return null when 401 occurs and no refresh token is available")
      • test("should return null without logout when 401 occurs and no session exists for refresh")
      • test("should cover final return path when 401 error has response and no refresh token")

III. Error Handler Tests: errorHandler.test.ts

These tests validate the error handling utilities used across the auth system to ensure consistent error processing.

Test Hierarchy:

  • describe("errorHandler")
    • handleApiError:

      Tests for the API error handling function.

      • test("should throw server-provided error message")
      • test("should fall back to axios error message when no server message")
      • test("should throw generic message when no meaningful message exists")
      • test("should throw generic message when not an Axios error")
      • test("should throw generic message for non-axios error")
    • withErrorHandling:

      Tests for the error handling wrapper function.

      • test("should return the result when function succeeds")
      • test("should handle errors from async functions")
      • test("should handle errors from sync functions")
